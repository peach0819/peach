整体逻辑

以下逻辑仅统计 BU_ID = 0 或者 PP订单,   且 SALE_DC_ID = -1 即海拍客订单.  且 item_dc_id = 0 即 海拍客供货.  除此以外其他业务不统计. 

PS: SALE_DC_ID = -1 是数仓的习惯. 具体实现以实际为准

公式

1) 当订单来源 不为 微小店 且 不为 "OMO B2B订单"时 且 不为 "SAAS订单" 时 (V2修改)

GMV = 门店实付金额(pay_amount) + 优惠券金额(coupon_amount)

2) 当订单来源 为 微小店 时

GMV = 门店实付金额(pay_amount) + 优惠券金额(coupon_amount)

微小店的 pay_amount依然是 门店支付金额. 

3) 当订单来源 为 "OMO B2B订单" 时

GMV = 门店实付金额(pay_amount) + 优惠券金额(coupon_amount)

注意! OMO订单这里只统计 B2B订单(bu_id=0) ,不统计 B2C订单(bu_id=51) ,避免双算. 

OMO订单的 pay_amount依然是 门店支付金额. 

4) 当订单来源 为 "SAAS订单" 时 (V2新增)

GMV = B端门店实付金额( ??? ) + 优惠券金额(coupon_amount)

例如： 海拍客售价：100  贝莱加价：20 贝莱门店支付金额：120  则：  GMV：100


注意! 目前SAAS订单里面的 pay_amount, 实际上是包含了 SAAS分销的利润的SAAS加价.   这部分加价的利润完全是给 SAAS那边的分销商的,跟海拍客无关. 所以不统计. 

在早期的需求 聆音的demo 里面,已经描述了 要把SAAS的改成 S2B的金额.  但当时没有做. 本次需求保证该需求实现.
