大商家组 : 触达流程&指标

Created by tao.zheng8833, last modified on 2023-01-29

背景（业务需求）

一个触达千人千面算法任务，圈选了5W门店时，在最复杂的策略场景下，需要能够在5分钟内开始进行第一批发送，同时分批发送之间不能存在间隔

概念

******************************

解释

**********************************************************************************************************************

千人千面任务触达任务中，圈选的每一家门店发送的内容，都根据门店个性化需求推荐
算法策略由ai-match提供，不同算法策略计算出的门店推品不同，耗时不同，对下游qps影响也不同
批次touch-core的实例，目前设定1000家门店为一批，一个5W家门店的触达任务，会被分成50个批次
分批发送目前触达中心发送机制为，按批次给算法数据，算法计算完一批，触达中心这边就开始进行这一批次的发送
分批发送之间不存在间隔

对于一个批次的门店（1000家），需要在第一批发送结束之前，第二个批次已经完成计算，即算法速率不能小于发送速率

5分钟内开始第一批发送第一批门店（1000家），要求任何策略的场景下，算法都能在5分钟内返回算法数据

触达中心系统计算阶段touch-center调用dmp获取任务门店

touch-center -> dmp

无qps要求，单个接口dubbo调用并返回，接口rt 3s以内
touch-center根据获取到的门店，分批调用算法ai-match，通知批次内门店信息以及分批信息touch-center -> ai-match无qps要求，目前1000家门店为一批，dubbo调用ai-match并实时返回
算法开始计算，调用mall-server获取门店召回池商品价格信息ai-match -> mall-server算法qps目前为15，具体到mall-server的qps根据策略不同存在不同，从压测结果看目前最高比例为1:3，具体策略qps比例明细需要咨询青云子
mall-server调用icp获取商品价格mall-server -> icp算法qps为15的情况下，ai-match → mall-server → icp的链路下，icp：ai-match的比例从压测结果来看最高为1:14，即ai-match为15，icp为210
算法根据获取到的门店商品价格，进行策略的精排、重排ai-match无qps要求，算法（ai-match、ai-service）内部实现
算法一批计算完成后，发送mq消息给touch-center, touch-center开始进行明细生成touch-center无qps要求，touch-center内部实现
明细生成结束，实例启动touch-center -> touch-core整体时长，1000家门店，需要在5分钟内完成系统计算阶段，具体时长主要取决于算法计算时长；从之前的压测表现看，在算法15qps前提下，icp210的qps几乎已经达到瓶颈，再提高qps时，icp的pod的cpu会飙升，存在风险。目前1000家门店在奶粉价格表策略（最复杂策略）下，算法常规计算时长为3分钟，能够满足业务需求的5分钟内开始第一批发送
触达中心发送阶段touch-core对已生成明细，并且已启动的实例进行推送touch-core

目前发送速率为单个企微，1分钟发送5家门店的消息；

目前一批1000家门店，单个企微分配到需要发送的最大场景为30家门店（库内5W家门店，单企微最大好友数为1500，比例为1000:30），按照发送速率需要6分钟内发送完成； 而目前的算法为3分钟计算完1000条，能够满足分批发送之间不存在间隔


现状

目前千人千面的触达任务，通过分批机制，能够满足业务需求的发送速率以及算法计算速率，同时算法计算阶段的主要瓶颈为icp价格获取（会员价格差异化项目上线后出现的），

之前在icp下线部分pod的压测中，跟八戒咨询过是否还有优化空间，八戒反馈目前icp由于资源限制，无法大幅度优化qps，如果后续icp能够支撑的qps能够提升，算法这边的qps随时可以上调，整体算法计算能够更快
