微商组的任务门店数据量查询

每个任务门店数  展开源码 
with detail as ( 
    SELECT  * FROM ytdw.dwd_touch_task_detail_d WHERE dayid = '${v_date}'
),

instance as (
    SELECT * FROM ytdw.dwd_touch_task_instance_d WHERE dayid = '${v_date}' AND channel_type = 1 AND instance_status = 3
)

SELECT instance.biz_id as task_id, instance.start_time, count(distinct detail.shop_id) as shop_cnt
FROM detail 
INNER JOIN instance ON detail.task_instance_id = instance.id 
WHERE substr(instance.start_time, 1, 8) = '${v_date}'
AND instance.creator IN (
    'f187b471db2348acb0b14e273a504e48',
    '5032d8424982403387372e3a57edf2fb',
    'f5140192d272427a85ceb3ab7ea3551e'
)
group by instance.biz_id, instance.start_time
order by instance.start_time

可以看到存在大量任务，且部分任务门店数接近2W

按照门店*品的数量计算

 展开源码 
with detail as ( 
    SELECT  * FROM ytdw.dwd_touch_task_detail_d WHERE dayid = '${v_date}'
),

instance as (
    SELECT * FROM ytdw.dwd_touch_task_instance_d WHERE dayid = '${v_date}' AND channel_type = 1 AND instance_status = 3
),

item_cnt as ( 
    SELECT task_id, sum(size(split(get_json_object(content_info, '$.content'), '#'))) as cnt
    FROM ytdw.dwd_touch_task_content_d 
    lateral view explode(split(regexp_replace(replace(REPLACE(content_template, '[', ''), ']',''),'\},','\}\;'),'\;')) tmp as content_info
    WHERE dayid = '20230712' 
    AND get_json_object(content_info, '$.type') = 1
    group by task_id
),

temp as (

SELECT instance.biz_id as task_id, instance.start_time, count(distinct detail.shop_id) as shop_cnt, item_cnt.cnt, count(distinct detail.shop_id)*item_cnt.cnt as total
FROM detail 
INNER JOIN instance ON detail.task_instance_id = instance.id 
INNER JOIN item_cnt ON instance.biz_id = item_cnt.task_id
WHERE substr(instance.start_time, 1, 8) = '${v_date}'
AND instance.creator IN (
    'f187b471db2348acb0b14e273a504e48',
    '5032d8424982403387372e3a57edf2fb',
    'f5140192d272427a85ceb3ab7ea3551e'
)
group by instance.biz_id, instance.start_time, item_cnt.cnt
order by instance.start_time
)

SELECT substr(start_time, 1,10) as start_time, sum(total) as total
FROM temp
group by substr(start_time, 1,10)
order by substr(start_time, 1,10)


结论：

如果按照千人千面模式，

实时调用成本：

每个门店获取的价格因子都需要单独调用

按照1小时135W店品计算，每秒钟需求店品计算量 = 1350000/3600 ≈ 375


目前导购给出的qps为5，分批最大品数量为20（如果为200个品，等于10次调用）

1分钟集群能够处理的店品数量：

1）如果按单品计算，每分钟 60*5 = 300个店品, 一小时能处理 18000个店品

2)  如果按多品-200品算，一小时只能处理18000/10 = 1800个店


按照目前多品的门店数来看，多品基本没法支持千人千面算量

单品可以支持一小时1W8以内的算量，所以全量门店的单品也基本上无法支持


投产比评估：

开发成本：

触达：

1、价格因子提前计算，额外表存储、价格计算后台任务开发、调用集群流控-3天

2、价格因子做实例分批（千人一面分批）-1天


导购：

1、单独接口限流，调用icp需要分批处理 - 1天


商品:

1、需要新增剔除会员价格逻辑接口，以保障性能 （原本icp老接口即可使用）- 1天


性能相关：

1、当前一天涉及发送的门店*品的数量在300W，平均时长一共12小时，如果按照提前处理，需要8小时处理300W数据量，平均每秒要求计算104的店品价，否则会影响任务数据准确性，同时对当前机器性能压力极大，存在影响底层价格计算的风险，会影响主站链路

2、任务计算时长会增长，同时价格实时性无法保证

3、当前千人千面任务算法侧调用导购 和 本次价格因子调用的qps会叠加，一旦出现重叠，qps将是翻倍的
